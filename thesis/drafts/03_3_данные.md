## 3.3. Модель данных

Модель данных спроектирована с разделением на on-chain и off-chain. Распределение выполнено по трём критериям: права собственности и голоса хранятся on-chain (неизменяемость), текстовые метаданные --- в реляционной БД (стоимость записи в блокчейне), для каталога используется реплика в PostgreSQL (быстрое чтение).

ER-диаграмма модели данных приведена на рисунке 3.6. Цветовая легенда отражает три категории хранения: **off-chain (PostgreSQL)** --- Users и Notifications; **гибридные (БД + блокчейн)** --- Projects и Milestones, где блокчейн хранит ключевые параметры, а PostgreSQL через Event Listener реплицирует их для UI и дополняет off-chain метаданными; **off-chain transactional** --- Investments, Votes, Transactions: каждая запись ссылается на транзакцию блокчейна через `tx_hash`, основные права определяются on-chain, off-chain запись существует для поиска и отчётности.

Рисунок 3.6

Модель включает семь сущностей: Users (учётные записи с bcrypt-хешем пароля и wallet_address), Projects (метаданные кампаний с contract_address), Milestones (этапы с report_uri), Investments (история инвестиций), Votes (голоса с весом = балансу токенов), Transactions (универсальный журнал операций) и Notifications (очередь уведомлений). Event Listener подписывается на WebSocket-канал узла и синхронизирует изменения при глубине не менее 12 блоков, что защищает от кратковременных реорганизаций цепи.

При потере off-chain БД критически важная информация (контракты, балансы, голоса) восстанавливается из блокчейна; теряются только вспомогательные метаданные, которые могут быть восстановлены из бэкапа.
