# Структура ВКР (v2 — с учётом дипломов на 10/10)
## «Разработка алгоритма и прототипа токенизированного фонда коллективного инвестирования»

---

## Введение (~5-6 стр.)

- **Актуальность** — рост краудфандинга, барьеры Web2-платформ (комиссии, география, непрозрачность), потенциал блокчейна и токенизации. Подкрепить цифрами из отчётов (Statista, CB Insights и др.)
- **Степень научной проработанности** — кто уже исследовал: Mollick (2014), Belleflamme et al. (2014), Ante (2021), Chen (2023) и др. Указать, какой пробел остаётся (нет комплексного решения для малых проектов с токенизацией)
- **Цель** — разработать алгоритм и прототип токенизированной краудфандинговой платформы для коллективного инвестирования в малые проекты
- **Задачи** (8 штук, по аналогии с Петросян):
  1. Проанализировать текущее состояние рынка краудфандинга и существующие Web2-решения
  2. Провести обзор научной литературы по токенизации активов и блокчейн-краудфандингу
  3. Выявить и систематизировать ключевые проблемы существующих платформ (диаграмма Исикавы)
  4. Проанализировать заинтересованные стороны и их требования
  5. Спроектировать бизнес-процессы платформы (BPMN AS-IS / TO-BE)
  6. Разработать алгоритм работы токенизированного фонда и логику смарт-контрактов
  7. Спроектировать архитектуру и модели данных информационной системы
  8. Реализовать прототип платформы и оценить его эффективность
- **Объект** — процесс коллективного инвестирования в малые проекты
- **Предмет** — информационная система (платформа), обеспечивающая токенизированный краудфандинг
- **Методы исследования** — сравнительный анализ, бенчмаркинг, причинно-следственный анализ (Исикава), моделирование бизнес-процессов (BPMN), анализ заинтересованных сторон, концептуальное и логическое моделирование данных, прототипирование
- **Научная новизна** — алгоритм токенизированного фонда, решающий проблемы Web2-краудфандинга для малых проектов
- **Практическая значимость** — готовый прототип платформы, пригодный к внедрению
- **Структура работы** — краткое описание содержания каждой из 3 глав (как у Петросян)

## ГЛАВА 1. АНАЛИЗ ПРЕДМЕТНОЙ ОБЛАСТИ И ВЫЯВЛЕНИЕ ПРОБЛЕМ (~25-30 стр.)

### 1.1. Краудфандинг: понятие, виды и модели (~4-5 стр.)
- Определение краудфандинга, его виды (reward-based, equity, donation, lending)
- Модели взаимодействия участников (платформа — автор проекта — инвестор)
- Роль краудфандинга в финансировании малых проектов и стартапов
- *Обзор литературы с цитатами из статей 2021+*

### 1.2. Анализ рынка краудфандинговых платформ (~5-6 стр.)
- Обзор Web2-платформ: Kickstarter, Indiegogo, GoFundMe, Boomstarter, Planeta.ru
- **Сравнительная таблица** (комиссии, география, механизмы защиты, модели финансирования, объёмы)
- Статистика рынка, тренды роста (с графиками)
- Модель окружения краудфандинговой экосистемы (по аналогии с Петросян)

### 1.3. Выявление и анализ ключевых проблем (~5-6 стр.)
- **Диаграмма Исикавы** — причинно-следственный анализ проблем Web2-краудфандинга (как у Петросян)
- Систематизация проблем по категориям:
  - Финансовые (высокие комиссии 5-12%, задержки выплат, валютные ограничения)
  - Географические (ограничения по странам, проблемы с KYC)
  - Прозрачность (непрозрачность расходования средств, мошенничество)
  - Ликвидность (невозможность выйти из проекта досрочно, нет вторичного рынка)
- Описание типичных конфликтных ситуаций (по аналогии с Петросян — конфликты интересов стейкхолдеров)

### 1.4. Блокчейн и токенизация активов (~4-5 стр.)
- Основы блокчейн-технологии, смарт-контракты
- Понятие токенизации, стандарты токенов (ERC-20, ERC-721, ERC-1155)
- Преимущества токенизации для коллективного инвестирования
- *Обзор литературы с цитатами*

### 1.5. Обзор существующих блокчейн-решений (~4-5 стр.)
- Обзор крипто-платформ (ICO/IDO-лаунчпады, DAO-фонды, DeFi-краудфандинг)
- **Сравнительная таблица** Web2 vs Web3 решений
- Выявление незанятой ниши — токенизированный краудфандинг для малых проектов

### 1.6. Анализ заинтересованных сторон (~3-4 стр.)
- Идентификация стейкхолдеров: инвесторы, авторы проектов, платформа, регуляторы, аудиторы
- **Матрица влияния/интереса** (Power-Interest Grid)
- Требования и ожидания каждой группы стейкхолдеров (таблица)
- Потенциальные конфликты интересов и способы их разрешения

### 1.7. Выводы по главе 1 (~1 стр.)
- Формулировка проблемы → обоснование необходимости разработки платформы
- Описание концепции целевого решения (краткое)

## ГЛАВА 2. ПРОЕКТИРОВАНИЕ ПЛАТФОРМЫ И РАЗРАБОТКА АЛГОРИТМОВ (~25-30 стр.)

### 2.1. Описание концепции платформы (~3-4 стр.)
- Value Proposition — какую ценность платформа даёт каждому стейкхолдеру
- Бизнес-модель (источники дохода, структура комиссий)
- Ключевые принципы решения (по аналогии с Петросян):
  - Прозрачность через блокчейн
  - Токенизация долей участия
  - Автоматизация через смарт-контракты
  - Защита инвесторов (milestone-based funding)

### 2.2. Моделирование бизнес-процессов AS-IS (~5-6 стр.)
- **BPMN-диаграмма** текущего процесса краудфандинга (Web2)
- **SIPOC-диаграмма** — входы/выходы процессов
- Описание ролей, этапов, точек принятия решений
- Выявление узких мест и неэффективностей (привязка к диаграмме Исикавы из Главы 1)

### 2.3. Моделирование бизнес-процессов TO-BE (~5-6 стр.)
- **BPMN-диаграмма** целевого процесса с токенизацией
- Сравнение AS-IS и TO-BE: что улучшается (таблица)
- Описание ключевых подпроцессов:
  - Создание проекта и выпуск токенов (Token Generation Event)
  - Процесс инвестирования (покупка токенов)
  - Управление средствами через смарт-контракт (эскроу)
  - Голосование инвесторов за разблокировку средств (milestone)
  - Возврат средств / распределение прибыли

### 2.4. Разработка алгоритма работы токенизированного фонда (~5-6 стр.)
- **Блок-схема** основного алгоритма работы фонда
- Алгоритм создания и верификации проекта
- Алгоритм сбора средств (пороги, дедлайны, refund при неуспехе)
- Алгоритм milestone-based распределения средств
- Описание конфликтных ситуаций и их разрешение (по аналогии с Петросян):
  - Конфликт: проект не достиг цели → автоматический refund
  - Конфликт: инвестор хочет выйти досрочно → вторичный рынок токенов
  - Конфликт: автор проекта нарушает milestone → голосование инвесторов
  - Конфликт: подозрение на мошенничество → механизм заморозки

### 2.5. Спецификация требований к системе (~4-5 стр.)
- **Функциональные требования** (таблица: ID, описание, приоритет)
- **Нефункциональные требования** (безопасность, масштабируемость, отказоустойчивость)
- **User Stories** для ключевых сценариев
- **Use Case диаграмма** — основные варианты использования

### 2.6. Моделирование данных (~4-5 стр.)
- **Инфологическая модель** (ER-диаграмма: сущности — проект, инвестор, токен, транзакция, milestone)
- **Даталогическая модель** (структура таблиц / структур данных в смарт-контрактах)
- Описание ключевых сущностей и связей между ними

## ГЛАВА 3. АРХИТЕКТУРА СИСТЕМЫ И РЕАЛИЗАЦИЯ ПРОТОТИПА (~20-25 стр.)

### 3.1. Выбор технологического стека (~3-4 стр.)
- Выбор блокчейн-сети (Ethereum / Polygon / BSC — обоснование)
- Язык смарт-контрактов (Solidity)
- Фронтенд (React + Web3.js/ethers.js)
- Бэкенд (Node.js / Python — если нужен off-chain компонент)
- База данных (PostgreSQL для off-chain данных)
- Обоснование каждого выбора

### 3.2. Архитектура решения (~4-5 стр.)
- **Диаграмма компонентов** (ArchiMate или UML)
- Слои архитектуры: блокчейн-слой, API-слой, UI-слой
- Структура смарт-контрактов (Factory pattern, Proxy pattern)
- Взаимодействие компонентов (sequence diagram для ключевых сценариев)

### 3.3. Реализация прототипа (~5-6 стр.)
- Реализация смарт-контрактов (ключевые фрагменты кода с пояснениями)
- Реализация фронтенда (скриншоты интерфейса)
- Реализация основных сценариев:
  - Создание проекта
  - Инвестирование
  - Голосование за milestone
  - Вывод средств

### 3.4. Тестирование прототипа (~3-4 стр.)
- Тестовые сценарии (таблица: сценарий, ожидаемый результат, фактический результат)
- Unit-тесты смарт-контрактов
- Демонстрация работы на тестовой сети (скриншоты транзакций)

### 3.5. Оценка эффективности решения (~4-5 стр.)
- **Сравнительная таблица** с Web2-аналогами (комиссия, скорость, прозрачность, география, ликвидность)
- Ожидаемые эффекты (по аналогии с Петросян — количественные метрики)
- **Ресурсно-календарный план** проекта (диаграмма Ганта)
- Оценка затрат на развёртывание (gas fees, хостинг, поддержка)
- Ограничения и перспективы развития
